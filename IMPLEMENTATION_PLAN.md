# Kwork-бот: план реализации (ответы на заявки биржи)

> ⚠️ Автоматизация откликов может нарушать пользовательское соглашение Kwork (риск блокировки аккаунта). Использовать осознанно.

## Архитектура

```
┌─────────────────────────────────────────────────────────────┐
│ 1. ЧТЕНИЕ ЗАЯВОК  — REST API  (api.kwork.ru)                  │
│    POST /signIn   → token, uad                                │
│    POST /projects → список заявок (фильтры: рубрики, бюджет,  │
│                     ключевые слова, страница)                 │
├─────────────────────────────────────────────────────────────┤
│ 2. ФИЛЬТР/ДЕДУП    — своя логика (бюджет, ключевые слова,     │
│                      «не отвечать дважды»)                    │
├─────────────────────────────────────────────────────────────┤
│ 3. ОТПРАВКА ОТКЛИКА — веб kwork.ru                            │
│    Вариант A (надёжно): Playwright открывает                  │
│       kwork.ru/new_offer?project=<id>, заполняет форму,       │
│       жмёт «Предложить».                                      │
│    Вариант B (быстро): HTTP-replay                            │
│       POST kwork.ru/api/offer/createoffer (FormData)          │
│       с cookie сессии + csrftoken.                            │
└─────────────────────────────────────────────────────────────┘
```

Гибрид: REST хорош для чтения списка (быстро, фильтры, пагинация), а **создание отклика есть только в вебе** — нативного API в APK нет.

## Эндпоинты

### Чтение (REST, `https://api.kwork.ru/`)
- `POST /signIn` — поля `login`, `password`, `uad` (device id), `device`. Ответ: `token`.
- `POST /projects` (getWorkerProjects) — `token`, `uad`, фильтры `categories`, `price_from`, `price_to`, `query`, `page`. Заголовки: `Authorization`, `Cookie`.

### Запись (веб, `https://kwork.ru/`)
- `POST /api/offer/createoffer` — **создать отклик**. FormData:
  - `wantId` = id заявки
  - `offerType` = `custom`
  - `description` = текст (≥150 симв.)
  - `kwork_duration` = **число дней** (целое из `durations.basic` = `[1,2,3,4,5,6,7,10,14,21,30,60]`)
  - `kwork_price` = сумма (что платит покупатель)
  - `kwork_name` = заголовок оффера
- Заголовки: `X-Requested-With: XMLHttpRequest`, cookie сессии. `csrftoken` — со страницы (`window.csrftoken`), для надёжности и в теле.
- `POST /api/offer/editoffer` — изменить существующий отклик (тот же payload).

## Рекомендация: Вариант A (Playwright)

Почему: форма — React/Vue-SPA с валидацией текста (`/api/validation/checktext`), авто-портфолио, vue-select сроком и CSRF. Браузер всё это делает «как человек» и устойчив к мелким изменениям вёрстки. HTTP-replay быстрее, но ломается при смене csrf-логики/полей и легче детектится антиботом.

Гибрид на практике:
1. Один раз залогиниться в браузере (Playwright, persistent context) → сохранить cookie/`storage_state`.
2. Список заявок — через REST (быстро) ИЛИ парсить `kwork.ru/projects`.
3. На каждую подходящую заявку — открыть `new_offer?project=<id>`, заполнить, отправить.

## Структура кода

```
kwork_bot/
  config.py          # креды, фильтры (бюджет/ключевые слова), тексты, задержки
  device.py          # генерация uad (SHA-1 от Build.* + android_id + ts)
  api_client.py      # REST: sign_in(), get_projects()
  filters.py         # отбор заявок + дедуп (хранить отвеченные id)
  offer_text.py      # генерация текста предложения (шаблоны/подстановки)
  submit_playwright.py  # Вариант A: заполнение формы и сабмит
  submit_http.py        # Вариант B: POST /api/offer/createoffer
  state.json         # отвеченные заявки (антиповтор)
  main.py            # цикл: poll → filter → submit → sleep
```

## Управление сессией
- REST: `token` живёт долго; при 401 — повторный `/signIn`. `uad` генерируется один раз и переиспользуется.
- Веб: `storage_state.json` Playwright (cookie). При разлогине — повторный логин-флоу (учесть капчу/SMS).

## Анти-бан / лимиты (обязательно)
- Случайные задержки между откликами (десятки секунд–минуты), не работать 24/7.
- Лимит откликов в час/день, учёт коннектов (списывается 1 на отклик).
- Уникализировать текст под каждую заявку (не один и тот же шаблон).
- Останавливаться при ошибках/капче, а не долбить повторно.
- Дедуп: не отвечать на одну заявку дважды.
